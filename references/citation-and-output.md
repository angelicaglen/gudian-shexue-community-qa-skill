# 证据、引用与输出规范

## 基本目标

引用不是装饰，而是让读者知道结论从哪里来。这个 skill 的默认体验应接近“智能体知识库”：回答自然，但事实边界清楚，关键判断可复核。

## 引用优先级

1. 正式章程、规则、公示文件。
2. 已复核的古籍、论文、图像、档案。
3. 已复核的实测数据。
4. 已整理并通过审核的内部讨论。
5. 原始聊天记录、口述、现场笔记。
6. 待核材料。

低优先级材料不能伪装成高优先级结论。

## 引用格式

API 返回的 `citation` 字段优先级最高。若 API 已返回 citation，直接使用，不自行重写编号。

常用格式：

- 单篇引用：`据《文件标题》【P003】……`
- 有文件编号：`据《文件标题》（KB-YYYYMMDD-0001，【P003】）……`
- 无段落号：`据《文件标题》“章节名”……`
- 多来源：分别列出来源，例如：章程部分见《文件A》【P002】；实测部分见《文件B》【P004】。
- 无可靠来源：必须说 `当前知识库没有可复核来源支持这个说法。`

## 默认回答：自然表达 + 末尾引用

普通问题不要机械堆引用。先把话说清楚，再在末尾给依据。

普通问题也不要机械暴露“缺了哪些库”。`missing_categories` 是内部边界提示，不是用户默认要看的内容。只有当缺失材料会改变核心结论时，才明确写出不足。

```text
结论：……

解释：……

依据：《文件标题》（KB-...，【P...】）。
```

短回答可以写成：

```text
结论是……。简单说，……。依据：《文件标题》（KB-...，【P...】）。
```

## 严谨引用输出

用户要求正式、可公开、要依据、要逐条引用时使用：

```text
结论：……

依据：
- 《文件A》（KB-...，【P...】）：……
- 《文件B》（KB-...，【P...】）：……

补充：……
```

严谨引用不是后台调试。不得输出 API 字段、检索路径、命中/未命中记录、reasoning 层级或 skill 工程说明。

## 争议问题输出

```text
这件事需要分开看。

规则依据：……

事实边界：……

可以这样回应：……
```

先降温，不回怼；优先说明规则背后的安全、社群治理或考据理由。

## 无足够资料输出

只在以下情况使用这个模式：

- API 成功，但核心问题没有任何可靠事实层材料。
- 用户问的是古籍出处、实测数据、器材参数、活动是否允许等强事实问题，而对应分类缺失。
- 如果不说明不足，用户会误以为这是知识库定论。

不要因为有某个辅助分类缺失，就把普通问答整体改写成“资料不足报告”。

```text
当前知识库没有足够材料支持这个判断。

已有线索：……

还需要补充：……
```

如果只能基于常识或临时材料回答，必须标注为“推测”或“待核”。

## 社媒或对外文案

```text
【正文】
……

【依据】
- 《文件标题》（KB-...，【P...】）
```

如果用户要求只输出正文，可以不显示“依据”区块，但事实性判断仍必须来自可追溯材料。

## 可信度规则

- `confidence: 高`：可以直接作为依据。
- `confidence: 中`：可引用，但说明是整理、复盘或二次材料。
- `confidence: 原始`：只能表述为“原始记录中有人提出过”或“原始记录显示”，不能直接作为定论。
- `confidence: 待核`：只能作为线索，不能作为结论依据。

## 隐私等级规则

- `privacy_level: 公开`：可正常引用标题、段落和内容。
- `privacy_level: 内部`：只能概括，不逐字引用，不输出群聊原句、成员姓名或内部争论细节。
- `privacy_level: 脱敏`：必须替换个人、场馆、联系方式等可识别信息。
- `privacy_level: 禁止外发`：不得引用具体内容，只能说明“这部分涉及内部或未授权材料，不能作为公开回答依据”。

## 冲突处理

当文献、实测、社群规则冲突时，按问题场景定标：

- 问历史事实或古人做法：以考据为主，数据只能作为现代复原参考。
- 问当下如何使用或选择器材：以数据、安全和器材规范为主，考据提供背景。
- 问联盟活动是否允许：以章程、公示文件和活动规则为准。
- 问价值判断或对外解释：以社群原则和文化表达为主，再补充考据。

不得用单一材料压倒所有场景。
